# Lucid Linux Alpha 0.2 Design

Date: 2026-05-21
Status: Draft for user review

## Goal

Alpha 0.2 repositions the project from LucidOS to Lucid Linux and turns the current Debian/KDE foundation into a coherent, polished desktop distribution. The release should feel very close to macOS in layout, window rhythm, and first-run polish while remaining an original Linux distribution with original assets, truthful Debian-based metadata, and clear upstream attribution.

## Non-Goals

- Do not use Apple names, logos, wallpapers, icons, screenshots, or copied interface assets.
- Do not pretend the system is not Debian-based in package metadata, licenses, source notices, or `ID_LIKE`.
- Do not fork Hermes Agent unless wrapping and configuring upstream Hermes proves insufficient.
- Do not make Alpha 0.2 depend on network access during first boot.

## Branding

The user-facing name is **Lucid Linux**. The old LucidOS name should be replaced across the surfaces controlled by this repository:

- README, roadmap, build docs, security docs, and future release notes
- ISO volume, ISO application, publisher text, and build output names
- `/etc/os-release`, `/etc/lsb-release`, `/etc/hostname`, and Lucid config files
- desktop entries, launchers, welcome messages, shell banners, and terminal prompts
- SDDM, boot menu, Calamares branding, and first-run onboarding

The system should keep truthful compatibility metadata:

- `ID=lucid`
- `ID_LIKE=debian`
- base package licenses and upstream notices preserved
- Hermes Agent license and attribution included

## Visual Identity

Lucid Linux Alpha 0.2 should use a very close macOS-inspired desktop language without copying Apple assets.

Core visual traits:

- clean desktop with no random icons
- top panel for menu/status indicators
- centered bottom dock-style panel for pinned apps
- translucent dark glass surfaces
- rounded rectangular windows where KDE allows it
- left-side window controls
- soft shadows, blur, and restrained animations
- custom Lucid logo and custom wallpaper
- consistent Lucid Glass colors across SDDM, Plasma, Konsole, boot, and installer

The wallpaper should be newly created for Alpha 0.2. It should be original, inspectable, and high-resolution. A good direction is a cinematic Lucid landscape or abstract glass vista with enough quiet space for desktop icons and login text.

## Desktop Shell

KDE Plasma remains the desktop environment for Alpha 0.2.

Default desktop configuration:

- top panel with application menu, active app/task affordance if feasible, clock, network, audio, battery, and session controls
- centered dock-style panel with pinned apps: file manager, terminal, browser if installed, settings, software center if installed, Lucid Agent, and installer in live mode
- desktop icons hidden by default except a single live-session installer shortcut if needed
- Breeze Dark used as the stable base, with Lucid Glass color scheme layered on top
- KWin effects enabled for blur/translucency when supported
- window buttons arranged on the left
- terminal profile themed with Lucid colors, readable font, and Lucid ASCII welcome

If exact macOS-like window rounding cannot be achieved through stock Debian Bookworm KDE packages, Alpha 0.2 should still set the closest stable KDE-native options and document the limitation.

## Hermes Agent Integration

Hermes Agent from Nous Research is preinstalled in Alpha 0.2 and exposed to users as **Lucid Agent powered by Hermes**.

Integration approach:

- install upstream Hermes without renaming or hiding its license
- add Lucid wrapper commands and desktop entries
- preserve upstream executable paths where practical
- expose a user-facing `lucid-agent` command
- show the app as "Lucid Agent" in launchers and onboarding
- include Hermes version and attribution in docs/about text

First-login behavior:

- after installation, Lucid Agent should automatically open once and onboard the user
- onboarding should explain what Lucid Agent can do, what permissions it may request, and how to configure providers
- onboarding should never require an API key before the desktop is usable
- if Hermes has a suitable GUI/onboarding mode, use it
- if not, launch a polished Konsole-based onboarding flow as the Alpha 0.2 fallback
- use a per-user marker file so onboarding runs once per installed user

Live-session behavior:

- Lucid Agent is available from the dock/launcher
- first launch opens the same onboarding experience, adapted for live mode
- no persistent secrets should be stored in the live image

## Boot Experience

Alpha 0.2 should add Lucid-branded boot surfaces where live-build supports them.

Boot menu entries:

- Try Lucid Linux
- Graphical Install Lucid Linux
- Try Lucid Linux (safe graphics)
- Hardware tools or memory test if supported by the chosen bootloader stack

Branding targets:

- isolinux menu text/colors for BIOS boot
- GRUB/UEFI menu text/colors if override points are available
- boot splash background or logo where supported
- Lucid ASCII/ANSI wordmark for TTY, terminal welcome, and onboarding

The ASCII art should be attractive and restrained: a Lucid wordmark, Alpha 0.2 line, and short startup/status message. It should render correctly in a plain Linux console.

## Graphical Installer

Graphical installation should be a first-class option.

Requirements:

- desktop launcher named "Install Lucid Linux"
- no visible "Install Debian" launcher in the live desktop
- boot menu option for graphical install
- Calamares starts automatically for the graphical install path if feasible
- Calamares branding should use Lucid Linux name, logo, wallpaper, and slideshow where Debian's packaged settings permit it
- installer completion should make the installed system boot into the Lucid Linux themed desktop

If the Debian Calamares settings package prevents complete rebranding, the remaining Debian strings must be documented as known Alpha 0.2 limitations.

## First Boot And Onboarding

The installed system should feel intentionally Lucid on first login.

First-login flow:

- SDDM uses the Lucid theme by default once validated
- user logs in to a clean Lucid desktop
- Lucid Agent onboarding opens once
- welcome content offers installer status, docs, settings, and agent setup
- user can dismiss onboarding without configuring AI

Implementation should prefer systemd user units, XDG autostart, or KDE autostart files with a marker file in the user's config directory. The flow must not loop on every login.

## Custom Assets

Alpha 0.2 needs original assets:

- Lucid logo SVG
- Lucid wallpaper, preferably both SVG source and PNG render
- SDDM background/brand mark
- boot splash image if supported
- Calamares logo/slideshow images if supported
- desktop launcher icons for Lucid Agent and installer where needed

All assets must be original or permissively licensed and documented.

## Error Handling And Fallbacks

- If Hermes installation fails during build, the build should fail clearly rather than silently shipping a broken agent launcher.
- If the Lucid SDDM theme is missing or invalid, the system should fall back to Breeze and log the reason during build.
- If Calamares is unavailable in the target Debian suite, hide the graphical installer option and document the blocker.
- If custom bootloader files break boot, revert to live-build defaults before release.
- If KDE layout files are ignored by Plasma, keep the defaults harmless and document manual testing results.

## Testing

Minimum verification before Alpha 0.2 is called complete:

- build script completes and produces a Lucid Linux Alpha 0.2 ISO
- live ISO boots in VirtualBox or QEMU
- boot menu shows Lucid Linux entries
- live desktop does not show "Install Debian"
- SDDM theme loads or fallback is documented
- desktop uses top panel, dock-style panel, left window buttons, and Lucid wallpaper
- Konsole uses Lucid profile and ASCII welcome
- `lucid-agent` launches Hermes-backed onboarding
- graphical installer launches from boot option and desktop launcher
- installed VM boots with Lucid Linux identity
- post-install first-login onboarding opens once and does not repeat after dismissal
- `/etc/os-release` reports Lucid Linux with `ID_LIKE=debian`

## Open Implementation Questions

- Which Hermes installation method is most reliable for an offline ISO build?
- Does the Debian Bookworm Calamares package expose enough branding/slideshow override points?
- Which browser or software center, if any, should be pinned by default?
- Can the current build pipeline render and package PNG assets from SVG sources consistently on CI?

These are implementation questions, not design blockers. The plan should resolve them by inspecting package availability and live-build behavior before editing the ISO configuration.
